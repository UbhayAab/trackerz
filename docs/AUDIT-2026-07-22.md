# Full-stack audit — 2026-07-22

Triggered by: "the AI overview is so wrong… I told it I did not go to gym, it
checkmarked that I went to gym, and the next day it was showing that you went to
the gym yesterday."

Method: every finding below was checked against the **live production database**
(project `yyoewdcijplkhxleejtm`), not inferred from code alone. 10 parallel
auditors produced 178 candidate findings; each was then handed to an independent
agent instructed to *refute* it. 116 survived. This document records the ones
that were acted on.

---

## The gym bug, end to end

Two independent defects compounded into the exact symptom reported.

**1. The pipeline logged a workout for a capture that DENIED one.**

`lib/fan-out-expander.mjs` salvages a workout whenever a capture *mentions* the
gym. Mention is not occurrence. There was no negation check anywhere in the
codebase.

Live `ai_actions` evidence:

```
2026-07-05 10:01:47  create_workout_log_candidate  auto_applied  0.60
  {"description": "Did not go to gym bro", "_auto_expanded": true}
```

Five of the seven rows in `workout_logs` were created this way — including the
capture in which the owner *reported this very bug* ("I did not do my workout
yesterday, In the ai note here, it says I did workout, wth"). The bug report
reproduced the bug.

**2. Any workout row counted as training.**

`lib/jarvis-brief.mjs` → `jbCloseDay()`:

```js
var workoutDone = workouts.length > 0 || steps >= 10000;
```

It never looked at *what* the row said. So the phantom row set
`flags.workout = true`, `jbNextStreaks` rolled the gym streak, and the next
morning's brief reported a session the owner had explicitly denied.

**Fix.** New pure module `lib/negation.mjs` (mirrored into the edge function),
clause-scoped so mixed captures still work — "no gym but ate 6 eggs" denies the
gym and still logs the eggs. A denial now writes an explicit `status='skipped'`
row rather than nothing, so the day is still *answered* (the logging streak
survives, the evening nudge stops asking) without counting as training.
`workout_logs.status` is new; only `'done'` counts toward streaks and the
weekly-workout goal.

Verified against the deployed function (`scripts/verify-negation-live.mjs`):

```
PASS  "No gym today"                            -> skipped
PASS  "Did not go to gym bro"                   -> skipped
PASS  "in gym, did workout A, bench 3x10 60kg"  -> done
```

A related contributor: `GYM_CUE` in the edge function matched a bare `workout`
while `GYM_WORDS` in `lib/capture-intent.mjs` did not. The two halves of the
pipeline disagreed about what a workout capture even *is*. Now aligned.

## "You got zero sleep"

Every morning brief asserted `sleep_h: 0` and the voice model narrated it —
"sleep was just 0 hours, so rest is a priority today". **The app has never had a
sleep data source.** `jbCloseDay` initialised `sleepH = 0` and there was nothing
to overwrite it.

Zero is a measurement. Absence is not. `sleepH` is now `null` when nothing was
recorded, the voice prompt states explicitly that null means *not measured, say
nothing about it*, and there is now a real source: the `sleep_sessions` table
plus the Sleeping/Woke-up button.

Same class of bug, also fixed:

- `under_budget` was true on a day with **nothing logged at all**, growing a
  budget streak for days the owner never opened the app. A day with no data is
  not evidence of ₹0 spent.
- `facts.yesterday.workout_done` actually carried `workout_ok`, so a rest day
  with zero training was narrated as "workout done". Both are now exposed
  separately.
- Every Supabase read in the Jarvis function discarded its error and substituted
  an empty array, so a transient failure was indistinguishable from "the user
  logged nothing" — and got frozen into `habit_days` permanently. Reads now fail
  loudly instead of recording a fiction.
- `src/state/sync.js` did the same thing on the frontend (`.catch(() => [])` on
  every fetch), so a dropped connection rendered "Nothing logged yet" over real
  data.

## Why sign-in was impossible

| What | State found | Consequence |
|---|---|---|
| `external_google_enabled` | `false` | "Continue with Google" could never work |
| `external_github_enabled` | `false` | same |
| `site_url` | `http://localhost:3000` | every magic link / reset email pointed at a dev server |
| `uri_allow_list` | *(empty)* | any `redirectTo` was rejected and downgraded to the above |
| `auth.identities` | only `email` | no Google identity exists on the account |

On top of that, three code defects:

- `signInWithOAuth` resolves with `error: null` even for a disabled provider (it
  only builds a URL and navigates), so the `if (error) throw` branch was dead
  code and the button could never report a failure.
- `initAuth()` returned the localStorage "developer local test mode" session
  **before ever contacting Supabase**. One tap on "Continue locally" permanently
  shadowed the real account: the app looked signed in, every query ran as
  `local:<email>`, and the real data was invisible with no way back.
- There was no password-reset path anywhere in the app.

**Fixed:** `site_url` and the redirect allow-list corrected on the project
(`scripts/fix-auth-config.mjs`); the real session now always wins over local
mode, with a loud banner and a one-tap exit when local mode is active; providers
are probed via `/auth/v1/settings` (which *is* authoritative) and disabled ones
are removed from the card instead of ejecting the user to a Supabase error page;
a working "Forgot password?" → reset-link → set-new-password flow now exists.

Google sign-in itself still needs a Google Cloud OAuth client ID + secret pasted
into Supabase → Authentication → Providers. That cannot be done from this repo.

## Money: the same ₹80 counted three times

On 2026-07-09 the capture *"Just ate 20 rupees lays and 60 for 3 boiled eggs"*
produced ledger rows three times — ₹80, then ₹20+₹60, then ₹20+₹60 again — after
an edge-function transport error and a re-submit. Root causes: no idempotency key
anywhere in the capture chain, and a transport error being treated as "nothing
was written" when the function may already have committed.

Also fixed in this pass: `computeBudgetAlerts` iterated *every* row of the shared
`budgets` table, so a daily-protein goal of 160 was compared against rupee spend
— ₹500 of spending produced "daily budget exceeded by 212%". Weekly windows
started on Sunday here and ISO-Monday on the Money page.

The idempotency work is **not** done; see below.

## Also fixed

- `hydration_logs` and `wellness_logs` had **zero rows, ever** — no tool, no UI,
  no read path. Water now has one-tap buttons and an agent tool.
- A leftover E2E test account (`e2e_*@test.local`) was a live profile row, so
  every Jarvis slot ran the whole pipeline twice and fired Resend at an address
  that cannot exist.
- A money-only mention ("Also yesterday lunch paid 110") synthesized a food row
  with NULL calories — a blank meal beside the real one. A meal-slot word names
  *when*, not *what*; no dish, no row.
- Dead/lying UI: the Money page's four summary tiles were hardcoded zeros; its
  Import button fabricated results (`rows=128, mapped=93%`) without importing;
  the status pill rendered `~undefineds` on every capture; deleting a row or
  ticking a meal kept the UI change even when the write failed; the diagnostics
  page marked passing checks as FAIL.
- Unescaped `innerHTML` for user-typed capture text and AI-authored plan names.

## Known-remaining (not fixed in this pass)

1. **No idempotency key on capture.** Re-submitting still double-writes. Needs a
   content fingerprint carried from client to edge function, plus a server-side
   check before insert — not a client-only fix, and not a naive unique index
   (which would forbid genuinely repeated identical purchases and fail silently).
2. **Bank statement imports never become ledger entries**, so imported rows are
   absent from every money total.
3. **`merged_into` is never written** — the duplicate merge hard-deletes instead,
   losing the audit trail. The subset-sum matcher that would catch
   "₹80 bank vs ₹20+₹60 voice" is written but unreferenced.
4. **The app is a dead shell offline**: `supabase-client.js` statically imports
   supabase-js from esm.sh, and the service worker refuses to cache it.
5. **Push still delivers nothing** until notifications are enabled once on the
   device (the button now exists; it never did before).
6. **Email lands from `onboarding@resend.dev`** — Resend's shared sandbox sender.
   It is accepted (HTTP 200) but almost certainly filed as spam. Fixing properly
   means verifying a domain in Resend and setting `JARVIS_EMAIL_FROM`.
